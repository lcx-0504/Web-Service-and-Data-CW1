  - 三个核心过程：Web Crawling（爬虫跟随超链接爬取页面）→ Indexing（将词与 URL

  - Deep Web：无法被搜索引擎索引的部分（表单后、付费内容等），比 Surface Web
  大数个数量级（冰山比喻）

  Linked Data 部分：
  - 传统 WWW 的数据是非结构化的，适合人读但不适合机器处理
  - Linked Data 是在 Web 上发布和连接结构化数据的技术（用 RDF Turtle 等格式）

  工具： 课程使用 Django（Python Web 框架）+ Django REST Framework

  Unit 2: HTTP - The Workhorse of the Web（20+页）

  深入讲解 HTTP 协议的方方面面：

  - Web Resources：不仅是静态文件，也可以是动态生成的内容
  - URI：每个资源有唯一标识符，格式为 scheme://host/path
  - 客户端-服务器模型：Client 发 request，Server 返 response

  HTTP 消息格式：
  - 三部分：Start line + Headers + Body
  - Request 格式：<method> <URL> <version> + headers + body
    - method 如 GET, POST, HEAD 等
  - Response 格式：<version> <status> <reason-phrase> + headers + body

  Status Code 分类：

  ┌──────┬──────────────────────────────────────────────────┐
  │ 范围 │                       含义                       │
  ├──────┼──────────────────────────────────────────────────┤
  │ 1xx  │ 信息性                                           │
  ├──────┼──────────────────────────────────────────────────┤
  │ 2xx  │ 成功（如 200 OK）                                │
  ├──────┼──────────────────────────────────────────────────┤
  │ 3xx  │ 重定向                                           │
  ├──────┼──────────────────────────────────────────────────┤
  │ 4xx  │ 客户端错误（如 404 Not Found, 401 Unauthorized） │
  ├──────┼──────────────────────────────────────────────────┤
  │ 5xx  │ 服务器错误                                       │
  └──────┴──────────────────────────────────────────────────┘

  Header 类型： General headers / Request headers / Response headers / Entity
  headers

  Media Types (MIME)：
  - HTTP 用 MIME 类型标记数据格式（如 Content-Type: image/jpeg）
  - 来自 MIME（Multipurpose Internet Mail Extensions），HTTP 借用了这套标准

  ---
  Unit 3: Representation State Transfer (REST)（20+页）

  详细拆解 REST 的含义和约束：

  REST 三个词的含义：
  - Representational：数据在传输时的"表示形式"（与存储形式可以不同，如 CSV
  存储可以用 JSON 返回）
  - State：实体（客户端/服务器）的状态，即它"知道什么"
  - Transfer：表示从服务器传到客户端（或反过来），从而改变对方的状态

  REST 这个词常被滥用 —— 很多自称 RESTful 的 API 并不满足 Roy Fielding
  定义的全部约束。如今更常说"RESTful API"而非"RESTful Web Service"。

  四大 REST 约束：

  1. Client-Server Architecture：前后端分离，各自独立演进
  2.
  Statelessness（无状态）：每个请求必须包含所有必要信息，服务器不记住之前的请求
    - 优点：可见性好、可靠性高、实现简单
    - 缺点：重复数据、客户端控制力弱
  3. Cacheability（可缓存）：响应需标记是否可缓存，减少不必要的交互
    - 优点：提升性能
    - 缺点：缓存过期导致数据不一致
  4. Uniform Interface（统一接口），包含 4 个子约束：
    - (1) 资源标识：每个资源有唯一且稳定的标识符（URI）
    - (2) 通过表示操作资源：表示中的信息足以修改资源
    - (3) 自描述消息：消息携带接收方处理所需的全部信息（如 media type）

  ---
  Unit 4: Bridging the Semantic Gap（20页）

  这是最有趣的一个 Unit，讨论 REST 的第四个统一接口子约束 —— 
  HATEOAS（超媒体作为应用状态引擎）：

  核心问题：语义鸿沟（Semantic Gap）
  - REST
  客户端可以请求、解析、识别链接，但如何决定下一步该请求什么？这需要领域知识

  用迷宫（Maze+XML）做例子：
  - 定义了一个自定义媒体类型 application/vnd.amundsen.maze+xml
  - 迷宫被表示为互相连接的"cell"网络，每个 cell 是一个 HTTP 资源
  - 每个 cell 的 XML 表示包含 <link> 标签，标注了方向（east/west/north/south）
  - 客户端通过跟随 link 来"走迷宫"，每跟随一个 link 就改变了应用状态

  Link Relations（链接关系）：
  - 是与超媒体控件关联的字符串，解释"点击这个链接会发生什么"
  - 由 IANA 管理，约 60 个已注册（如 next, prev, author, nofollow 等）
  - 两种：registered（短字符串）和 extension（URL 形式）

  Fiat Standards（事实标准）：
  - 几乎所有现实中的 API 都是 fiat standard（一家公司的一次性设计）
  - 理想情况下应该在已有标准上加一层薄薄的 gloss

  迷宫 API 的完整结构：
  - Collection → Item（一个具体迷宫）→ Cell（迷宫中的格子）
  - 定义的 link relations：maze（进入迷宫）、start（起点）、east/west/north/sout
  h（方向）、exit（出口）

⏺ 逐个概念解释

  1. 数据模型 (Data Model)

  就是你的数据"长什么样"。用 Python 来类比，就是定义一个类：

  # 比如你做一个音乐 API，一个数据模型可能是：
  class Song:
      id = 1
      title = "Shape of You"
      artist = "Ed Sheeran"
      genre = "Pop"
      duration = 233  # 秒
      rating = 4.5

  在数据库里，这就对应一张表（table），每个字段是一列，每首歌是一行。

  作业要求"至少一个"，但想拿高分建议做 2-3 个相互关联的模型（比如
  Song、Artist、Playlist）。

  ---
  2. CRUD

  四种最基本的数据操作，几乎所有应用都围绕这四个动作：

  ┌────────────┬──────┬─────────────┬─────────────────────────────────┐
  │    字母    │ 操作 │  HTTP 方法  │       举例（以歌曲为例）        │
  ├────────────┼──────┼─────────────┼─────────────────────────────────┤
  │ C - Create │ 创建 │ POST        │ 添加一首新歌到数据库            │
  ├────────────┼──────┼─────────────┼─────────────────────────────────┤
  │ R - Read   │ 读取 │ GET         │ 查询歌曲列表 / 查看某首歌的详情 │
  ├────────────┼──────┼─────────────┼─────────────────────────────────┤
  │ U - Update │ 更新 │ PUT / PATCH │ 修改某首歌的评分                │
  ├────────────┼──────┼─────────────┼─────────────────────────────────┤
  │ D - Delete │ 删除 │ DELETE      │ 删除某首歌                      │
  └────────────┴──────┴─────────────┴─────────────────────────────────┘

  ---
  3. 数据库 (Database)

  持久化存储数据的地方。作业要求用 SQL 数据库，常见选项：

  - SQLite — 最简单，一个文件就是一个数据库，Django/FastAPI
  默认支持，适合这个作业
  - PostgreSQL — 更专业，想加分可以用
  - MySQL — 也行

  简单理解：Excel 表格的升级版，但支持用代码高效查询和操作。

  ---
  4. RESTful API

  课件 Unit 1-3 讲的核心内容。简单说就是一套设计风格，规定你的 API
  应该怎么组织：

  GET    /api/songs/        → 获取所有歌曲（Read）
  GET    /api/songs/42/     → 获取 id=42 的歌曲（Read）
  POST   /api/songs/        → 创建新歌曲（Create）
  PUT    /api/songs/42/     → 更新 id=42 的歌曲（Update）
  DELETE /api/songs/42/     → 删除 id=42 的歌曲（Delete）

  核心规则（来自课件 Unit 3）：
  - 用 URL 标识资源（/songs/42 就是第 42 首歌）
  - 用 HTTP 方法表示动作（GET 读、POST 创建、PUT 更新、DELETE 删除）
  - 无状态：每个请求独立，服务器不记得你之前做了什么
  - 返回 JSON 格式的数据

  ---
  5. GraphQL

  REST 的一种替代方案。区别在于：

  ┌──────────┬────────────────────────────┬──────────────────────────────────┐
  │          │            REST            │             GraphQL              │
  ├──────────┼────────────────────────────┼──────────────────────────────────┤
  │ 请求方式 │ 多个 URL，每个对应一个资源 │ 一个 URL，客户端指定要什么数据   │
  ├──────────┼────────────────────────────┼──────────────────────────────────┤
  │ 例子     │ GET /songs/42 返回固定格式 │ 客户端说"我只要 title 和 rating" │
  ├──────────┼────────────────────────────┼──────────────────────────────────┤
  │ 优点     │ 简单直观                   │ 灵活，避免多余数据               │
  ├──────────┼────────────────────────────┼──────────────────────────────────┤
  │ 缺点     │ 可能返回不需要的字段       │ 学习成本高                       │
  └──────────┴────────────────────────────┴──────────────────────────────────┘

  我的建议：用 REST 就好。 课程教的就是 REST，用 GraphQL
  还要额外论证为什么，增加工作量但不一定加分。除非你之前有 GraphQL 经验。

  第一层：你的 API（纯 HTTP 接口）

  你写了一个 FastAPI 应用，跑在 localhost:8000：

  GET  http://localhost:8000/api/foods/search?q=鸡蛋

  返回：
  {"id": 1, "name": "鸡蛋", "calories": 144, "protein": 12.6, "fat": 9.5}

  requests、curl。但 AI 不知道这个 API 存在，也不知道怎么用它。

  ---
  第二层：MCP Server（给 AI 一份"说明书"）

  MCP（Model Context Protocol）的核心思想很简单：用一种标准格式告诉 AI 
  "你有哪些工具可以用，每个工具接收什么参数，返回什么"。

  你写一个 MCP Server，本质上就是一个翻译层 / 中间人：

  # mcp_server.py
  from mcp import FastMCP

  mcp = FastMCP("Nutrition API")

  @mcp.tool()
  def search_food(query: str) -> str:
      """搜索食品数据库，返回匹配的食品及其营养信息"""
      # 内部调用你自己的 API
      response =
  requests.get(f"http://localhost:8000/api/foods/search?q={query}")
      return response.json()

  @mcp.tool()
  def log_meal(user_id: int, food_ids: list[int], quantities: list[float]) -> 
  str:
      """记录用户的一餐饮食"""
      response = requests.post("http://localhost:8000/api/meals/", json={...})
      return response.json()

  @mcp.tool()
  def get_daily_nutrition(user_id: int, date: str) -> str:
      """获取用户某天的营养摄入汇总"""
      response = requests.get(f"http://localhost:8000/api/users/{user_id}/daily-
  summary?date={date}")
      return response.json()

  MCP Server 做了什么？
  - 把你的 HTTP 端点包装成一个个"工具"（tool）
  - 每个工具有名字、参数描述、功能说明（那个 docstring 很关键，AI
  靠它理解工具用途）
  - 对外暴露一个标准的 MCP 协议接口

  ---
  第三层：AI 怎么用这些工具

  当一个 MCP 客户端（Claude Desktop、ChatBox 等）连接到你的 MCP Server
  时，流程是：

  1. 客户端启动时，问 MCP Server："你有哪些工具？"

  2. MCP Server 回答：
     - search_food(query: str) — "搜索食品数据库"
     - log_meal(user_id, food_ids, quantities) — "记录一餐"
     - get_daily_nutrition(user_id, date) — "获取每日营养汇总"

  3. 这些工具描述被注入到 AI 的上下文中（类似 system prompt）

  4. 用户说："我今天中午吃了两个鸡蛋和一碗米饭"

  5. AI 思考：我需要先搜索"鸡蛋"和"米饭"的营养数据
     → 调用 search_food("鸡蛋")
     → 调用 search_food("米饭")

  6. MCP Server 收到调用 → 转发给你的 API → 拿到结果 → 返回给 AI

  7. AI 拿到数据后继续：记录这顿饭
     → 调用 log_meal(user_id=1, food_ids=[1,2], quantities=[2,1])

  8. AI 最后总结："你午餐摄入了约 500 卡路里，蛋白质
  28g，建议晚餐多吃蔬菜补充纤维..."

  是不安全的。让我把目前工程化的几种做法讲清楚。

  核心矛盾

  REST 是无状态的（每次请求带 token 认证）
      +
  MCP 协议中，AI 不应该持有用户的敏感凭证
      =
  谁来管 token？

  目前主流的三种做法

  方案 1：MCP Server 持有 token（最常见）

  用户 → AI → MCP Server（内置 token）→ API

  MCP Server 作为一个可信中间层，启动时就配置好了认证信息。AI 完全不知道 token
  的存在：

  # mcp_server.py
  import os

  API_TOKEN = os.environ["API_TOKEN"]  # 从环境变量读取，不暴露给 AI

  @mcp.tool()
  def search_food(query: str) -> str:
      """搜索食品"""
      response = requests.get(
          "http://localhost:8000/api/foods/search",
          params={"q": query},
          headers={"Authorization": f"Bearer {API_TOKEN}"}  # MCP Server 
  自己加上
      )
      return response.json()

  AI 只管调用 search_food("鸡蛋")，完全不知道背后有认证这回事。

  适用场景：单用户，或 MCP Server 和 API 都是你自己的

  ---
  方案 2：OAuth 流程（多用户场景）

  用户 → 浏览器登录授权 → 获得 token → 存在 MCP Server 的 session 中
                                            ↓
                             AI → MCP Server（从 session 取 token）→ API

  流程：
  1. 用户在网页上登录（用户名密码 / OAuth）
  2. API 返回一个 JWT token
  3. Token 存在 MCP Server 的服务端 session 里（和用户会话绑定）
  4. AI 调用工具时，MCP Server 自动从 session 取出对应用户的 token

  AI 依然看不到 token。

  ---
  方案 3：API 本身不做认证（最简单）

  AI → MCP Server → API（无需认证）

  对于你的作业来说，这其实是完全可行的。理由：
  - 这是一个课程项目，不是生产环境
  - API 跑在 localhost，不对外暴露
  - 用户身份可以通过 URL 参数传递（/api/users/1/meals/）

  ---
  针对你的作业，我的建议

  分两层设计，务实为主：

  ┌─────────────────────────────────────────┐
  │ API 层：实现 JWT 认证                     │
  │                                         │
  │  POST /api/auth/register  → 注册        │
  │  POST /api/auth/login     → 返回 JWT    │
  │  其他端点需要 Bearer token               │
  │                                         │
  │  → 这部分展示你理解认证机制（加分）        │
  └─────────────────────────────────────────┘

  ┌─────────────────────────────────────────┐
  │ MCP Server 层：持有 token                │
  │                                         │
  │  启动时自动登录，获取 token               │
  │  所有工具调用自动带上 Authorization       │
  │  AI 完全无感知                           │
  │                                         │
  │  → 这是方案 1，简单且安全                │
  └─────────────────────────────────────────┘

  具体流程：

  # MCP Server 启动时
  class NutritionMCP:
      def __init__(self):
          # 启动时用预设账号登录，拿到 token
          resp = requests.post("http://localhost:8000/api/auth/login",
                               json={"username": "mcp_agent", "password":
  "xxx"})
          self.token = resp.json()["access_token"]

      @mcp.tool()
      def search_food(self, query: str) -> str:
          """搜索食品"""
          return requests.get(
              ".../api/foods/search",
              params={"q": query},
              headers={"Authorization": f"Bearer {self.token}"}
          ).json()

  先区分两个概念

  你说的 OAuth — 其实有两种场景

  场景 A：你的 API 作为 OAuth Provider（你自己发 token）
  用户 → 登录你的系统 → 你发 JWT token → 用 token 访问你的 API
  这其实就是 JWT 认证，不是严格意义上的 OAuth。

  场景 B：真正的 OAuth 2.0（借助第三方身份认证）
  用户 → "用 Google/GitHub 登录" → 跳转到 Google → 用户授权
  → Google 回调你的 API 带上 authorization code → 你换取 token
  → 拿到用户信息 → 创建本地账号 → 发你自己的 JWT

  场景 B 才是真正的 OAuth 流程，涉及你说的回调、授权码等。

  LangChain 介绍

  是什么

  LangChain 是一个 Python/JS 框架，帮你把 LLM
  和外部工具串联起来。核心就是解决：怎么让 AI 
  不只是聊天，而是能调用工具、查数据库、执行操作。

  核心概念

  # 没有 LangChain 时，你得自己写这些逻辑：
  # 1. 把工具描述发给 AI
  # 2. 解析 AI 的回复，判断它想调用哪个工具
  # 3. 执行工具
  # 4. 把结果喂回 AI
  # 5. 循环直到 AI 给出最终回答

  # 有 LangChain 时：
  from langchain.agents import create_tool_calling_agent
  from langchain_openai import ChatOpenAI
  from langchain.tools import tool

  @tool
  def search_food(query: str) -> str:
      """搜索食品数据库"""
      return requests.get(f".../api/foods/search?q={query}").json()

  @tool
  def log_meal(user_id: int, food_ids: list, quantities: list) -> str:
      """记录用户的一餐"""
      return requests.post(".../api/meals/", json={...}).json()

  # 一行搞定 agent
  llm = ChatOpenAI(model="gpt-4o")
  agent = create_tool_calling_agent(llm, tools=[search_food, log_meal],
  prompt=...)

  LangChain 帮你处理了上面的 1-5 步循环。

  但是 — 你需要 LangChain 吗？

  对于你的项目，我建议不用 LangChain。 理由：

  ┌──────────────┬────────────────────────────┬─────────────────────────────┐
  │              │         LangChain          │      直接用 SDK + MCP       │
  ├──────────────┼────────────────────────────┼─────────────────────────────┤
  │ 学习成本     │ 高，API 变动频繁，文档混乱 │ 低                          │
  ├──────────────┼────────────────────────────┼─────────────────────────────┤
  │ 依赖         │ 很重，一堆子包             │ 轻量                        │
  ├──────────────┼────────────────────────────┼─────────────────────────────┤
  │ 和 MCP       │ 有 MCP adapter 但不是原生  │ MCP 本身就是为此设计的      │
  │ 的关系       │                            │                             │
  ├──────────────┼────────────────────────────┼─────────────────────────────┤
  │ 调试         │ 黑盒，出了问题难排查       │ 透明                        │
  ├──────────────┼────────────────────────────┼─────────────────────────────┤
  │ 面试/答辩    │ "你为什么用 LangChain？"   │ "我用 MCP 标准协议"         │
  │              │ 难答                       │ 更有说服力                  │
  └──────────────┴────────────────────────────┴─────────────────────────────┘

  你已经有 MCP 了，MCP 本身就做了 LangChain Agent 做的事情（告诉 AI
  有什么工具、怎么调用）。再加一层 LangChain 是多余的。

  你的网页 AI 助手怎么实现（不用 LangChain）

  直接用 Anthropic/OpenAI 的 SDK，手动实现 tool calling：

  # 后端：处理 AI 对话的 endpoint
  @app.post("/api/chat")
  async def chat(message: str, user_token: str):

      # 定义 AI 可用的工具（和 MCP tools 一一对应）
      tools = [
          {
              "name": "search_food",
              "description": "搜索食品数据库，返回营养信息",
              "input_schema": {
                  "type": "object",
                  "properties": {
                      "query": {"type": "string", "description": "食品名称"}
                  }
              }
          },
          # ... 其他工具
      ]

      # 1. 发给 AI
      response = anthropic.messages.create(
          model="claude-sonnet-4-20250514",
          messages=[{"role": "user", "content": message}],
          tools=tools
      )

      # 2. 如果 AI 想调用工具
      if response.stop_reason == "tool_use":
          tool_call = response.content[1]  # AI 说要调 search_food("鸡蛋")

          # 3. 执行工具（调用你自己的 API）
          result = call_your_api(tool_call.name, tool_call.input, user_token)

          # 4. 把结果喂回 AI
          final = anthropic.messages.create(
              messages=[..., {"role": "tool", "content": result}],
              tools=tools
          )

          return final.content  # AI 的最终回答

      return response.content

  这比 LangChain 更直观，你完全理解每一步在做什么，答辩时也能清楚解释。

⏺ Swagger 是什么

  Swagger（现在官方名叫 OpenAPI）是一套 API 文档自动生成和测试的工具体系。

  核心思想

  你写了一个 API，Swagger 自动帮你生成一个漂亮的交互式网页，列出所有端点，别人可
  以直接在网页上点按钮测试你的 API：

  你写代码 → Swagger 自动扫描 → 生成文档网页

  FastAPI 自带 Swagger

  这就是我推荐 FastAPI 的重要原因之一。你什么都不用配，启动后直接访问：

  http://localhost:8000/docs      ← Swagger UI（交互式）
  http://localhost:8000/redoc     ← ReDoc（另一种样式，更适合阅读）

  举个例子，你写了这段代码：

  @app.get("/api/foods/search")
  async def search_food(
      q: str = Query(..., description="搜索关键词，如'鸡蛋'"),
      limit: int = Query(10, description="返回数量上限")
  ) -> list[FoodResponse]:
      """搜索食品数据库，返回匹配的食品及其营养信息"""
      ...

  Swagger 自动生成的页面就会展示：

  ┌─────────────────────────────────────────────────┐
  │  GET  /api/foods/search     搜索食品数据库...     │
  │                                                 │
  │  Parameters:                                    │
  │  ┌─────────┬────────┬──────────────────────┐    │
  │  │ Name    │ Type   │ Description          │    │
  │  ├─────────┼────────┼──────────────────────┤    │
  │  │ q*      │ string │ 搜索关键词，如'鸡蛋'    │    │
  │  │ limit   │ int    │ 返回数量上限（默认10）   │    │
  │  └─────────┴────────┴──────────────────────┘    │
  │                                                 │
  │  Response (200):                                │
  │  [{"id": 1, "name": "鸡蛋", "calories": 144}]  │
  │                                                 │
  │  [    Try it out    ]  ← 可以直接在网页上测试！    │
  └─────────────────────────────────────────────────┘

  对你的作业意味着什么

  作业要求交一份 API 文档 PDF。用 FastAPI 的话：

  1. 代码里写好类型注解和 docstring
  2. Swagger UI 自动生成
  3. 打开 /docs → 浏览器打印为 PDF → 交作业

  API 文档这一项几乎零额外工作量。

  如果用 Django，你得额外装 drf-yasg 或 drf-spectacular 来生成 Swagger，配置比
  FastAPI 麻烦很多。

  Swagger vs OpenAPI 的关系

  ┌───────────────────────┬────────────────────────────────────────────────┐
  │         名称          │                     是什么                     │
  ├───────────────────────┼────────────────────────────────────────────────┤
  │ OpenAPI Specification │ 一种描述 API 的 JSON/YAML 标准格式（规范本身） │
  ├───────────────────────┼────────────────────────────────────────────────┤
  │ Swagger UI            │ 把 OpenAPI 规范渲染成交互式网页的工具          │
  ├───────────────────────┼────────────────────────────────────────────────┤
  │ ReDoc                 │ 另一种渲染工具，排版更美观                     │
  └───────────────────────┴────────────────────────────────────────────────┘

  三者的关系就像：Markdown（规范） vs GitHub Pages（渲染器 A） vs Notion（渲染器
   B）。

  FastAPI 自动生成 OpenAPI 规范（/openapi.json），然后同时用 Swagger UI 和 ReDoc
   两种方式渲染给你。

  我的推荐

  后端：    FastAPI
             - 自带 Swagger（省掉 API 文档工作）
             - 异步性能好
             - 代码简洁，类型注解即文档
             - 报告中可以论证"选 FastAPI 而非课程教的 Django"展示独立思考

  数据库：  SQLite + SQLAlchemy
             - 零配置，答辩时不怕环境问题
             - SQLAlchemy 是行业标准 ORM

  前端：    React（如果你熟）或 纯 HTML+JS（如果你不熟前端框架）
             - 前端不是评分重点，能跑通就行

  MCP：     Python mcp 官方包
             - 和后端同语言，一个项目搞定

  AI SDK：  Anthropic SDK
             - 原生 tool calling，和 MCP 概念一致

⏺ 好问题，逐个回答：

  1. 饮食建议 → 交给 AI

  完全同意。/api/analytics/suggestions 这个端点删掉。AI 通过 MCP 拿到
  daily/weekly 的营养数据后，自己就能给出建议，比你硬编码规则灵活得多。

  2. 用户信息字段

  - 头像 (avatar_url)：GitHub OAuth 回调会自带，留着不费事，前端显示用
  - 年龄/体重/身高：这些是计算每日推荐摄入量（TDEE）的必要参数。比如一个 80kg
  的成年男性和一个 50kg
  的女性，推荐每日热量差很多。但如果你不想做推荐值计算，确实可以去掉
  - 饮食目标 (dietary_goal)：就是用来影响推荐值的。减脂的人推荐热量要低于
  TDEE，增肌的要高。如果推荐值计算交给 AI 做，这个字段 AI
  可以直接问用户，不用存数据库

  结论： 如果营养均衡评估（balance 端点）用固定的通用推荐值（比如成人每日
  2000kcal），那年龄/体重/身高/dietary_goal
  都可以去掉。如果你想做个性化推荐，就留着。

  我的建议：去掉这些字段，简化 User 模型。 个性化的部分让 AI
  在对话中问用户就好，不用持久化。

  3. 数据来源 (source 字段)

  同意去掉。只用 USDA 一个数据源，没必要加这个字段。

  4. 更新食品信息 (PUT /api/foods/{id})

  CRUD 里的 U。作业要求"完整的
  CRUD"，所以必须有。场景：管理员修正了某个食品的营养数据（比如热量录入错了）。

  5. 获取分类 (GET /api/foods/categories)

  这个确实不是必需的，删掉也行。它的用途是前端做个下拉筛选器。如果前端不做食品浏
  览页面（你的前端主要是聊天界面），这个端点没用。
